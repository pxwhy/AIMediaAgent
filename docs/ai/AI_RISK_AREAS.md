# AI 风险区域

## 高风险模块总览

当前项目的高风险点主要集中在：账号登录态、平台自动化、模型配置、智能筛选、Skills 注入、前端模块拆分。

## 前端模块拆分

风险：

- `provide/inject` key 使用普通 `Symbol` 时，HMR 或重复模块实例会生成不同 key，导致页面白屏。
- 源码目录混入构建生成的 `.js` 文件后，无扩展导入可能命中不同模块实例。
- 主页面拆分后共享上下文遗漏字段，导致子模块运行时访问空对象。

要求：

- 跨模块共享的注入 key 使用 `Symbol.for(...)` 保持运行时稳定。
- 拆分页面后必须验证开发服务页面渲染，不只依赖 `npm run build`。
- 子模块从共享上下文取值时，新增依赖要同步补到 `provide` 列表。

## 账号与登录态

风险：

- Cookie / storage_state 泄露
- 登录态失效导致同步或发布失败
- 同平台重复账号覆盖错误

要求：

- 敏感信息不能进入前端日志和模型 prompt。
- 登录态只能由账号模块维护。
- 账号创建和登录确认必须按 `platform + uid` 去重。

## 头条号自动发布

风险：

- 页面结构变化导致按钮定位失效
- 广告收益勾选项变化导致发布未完成
- 页面打开成功但实际未发布

要求：

- 必须保留发布前后日志与截图。
- 勾选“投放广告赚收益”逻辑要可回归验证。
- 平台返回异常时必须落失败状态，不能只看页面跳转。

## 小红书自动发布

风险：

- 发布按钮位于 Shadow DOM 内，普通选择器误判
- 只看可见 DOM 但实际 root 在 shadow tree
- 固定坐标点击在不同屏幕、缩放、窗口尺寸下不稳定

要求：

- 先验证宿主节点，再进入 ShadowRoot 查找按钮。
- 优先使用稳定 DOM 选择器。
- 不把固定分辨率点击作为正式方案。

## 发布诊断

风险：

- 失败时缺少可读日志
- 截图和平台返回不完整，导致难以定位
- 运行目录堆积导致排查混乱

要求：

- 每次发布要保留 `run_dir`、日志、截图、错误信息、平台返回。
- 前端详情查看必须可读，便于追问题。

## 模型配置

风险：

- Base URL 配错导致调用失败
- API Key 泄露
- 从 `.env` 自动带出旧配置，引发误解
- OpenAI-compatible 与供应商字段不一致

要求：

- 模型配置以页面填写结果为准，不偷偷回读 `.env` 混淆用户。
- 所有模型调用必须有 timeout、错误处理、usage 记录。
- 类似 `Invalid port: ':1'` 这类错误要原样暴露给用户排查。

## Agent 与 Skills

风险：

- Prompt 漂移导致输出结构不稳定
- Skill 目录扫描成功，但内容不符合约定格式
- Agent 绑定错误 Skill 后输出偏题

要求：

- Agent 输出要尽量结构化。
- Skill 目录结构要和约定保持一致。
- Skill 只参与 prompt 约束，不直接控制账号、发布或数据库。

## 智能筛选

风险：

- 仅靠模糊匹配导致素材误选
- 复盘、肖像、来源分类三者依据混乱
- 兜底逻辑过多掩盖真实判断不足

要求：

- 当前应优先基于复盘和账号肖像做显式判断。
- 匹配依据要能在结果中解释。
- 不明确时宁可少选，也不要伪造高置信度选择。

## 作品同步

风险：

- 平台列表里夹带栏目、草稿、收益页等非作品项
- 同步不到正文，只拿到标题
- 数据去重不稳定导致列表污染

要求：

- 同步时过滤无效作品项。
- 标题和正文要尽量都回收。
- 同步逻辑要按平台作品 ID 去重。
