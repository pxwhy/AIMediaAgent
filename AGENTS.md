# AIMediaAgent 协作约束

## 基本要求

1. 新需求在 coding 前先确认设计方案，不直接修改代码。
2. 默认使用中文回复。
3. 业务代码文件顶部必须增加“实现逻辑”注释。
4. 代码实现必须遵守本仓库的模块边界和代码风格要求。

## 代码风格要求

- Python 使用 `snake_case`。
- TypeScript / Vue 变量和函数使用 `camelCase`。
- 组件名、类型名、类名使用 `PascalCase`。
- 命名要语义化，避免无意义缩写和临时命名残留。
- 单个函数职责要清晰，超长函数优先拆分。
- 路由层不堆业务逻辑，业务逻辑进入 `services/`。
- 前端请求封装统一放 `apps/web/src/api/`。
- 新前端业务模块按职责建立目录，优先拆成 `index`、`add`、`edit` 等入口；列表、创建、编辑不要长期堆在同一个大组件里。
- 注释优先说明“为什么”，不要堆无效解释。

## 前端实现要求

- 查看、详情、编辑优先统一成弹框交互。
- 新增操作优先放在列表工具栏，点击后打开弹框表单，不在列表页内联展开大表单。
- 编辑操作优先打开弹框表单，避免在表格行内直接展开复杂编辑区。
- 新模块的列表页使用 `index` 入口，新增表单使用 `add` 入口，编辑表单使用 `edit` 入口；即使最终以弹框展示，也要按职责拆分组件。
- 弹框出现长内容时，优先让内容区滚动，头部信息固定，避免关键信息滚走。
- 列表页操作列位置、分页交互、状态展示尽量统一。
- 表格工具栏只保留操作类按钮和必要筛选控件，例如新增、批量删除、刷新、筛选；不要放说明性计数文本。
- 表格底部只放分页控件，不放“已选 N 条”“共 N 条”等说明计数；选择数量如确实必要，放到操作按钮状态或弹框确认文案中。
- 素材、复盘、肖像、诊断等详情查看不再回退成页面内联大面板。
- 工作台页面优先实用性，不做营销式布局。
- 样式改动优先和当前工作台风格保持一致。
- 已有页面存在成熟交互范式时，优先复用，不为单个页面临时发明新样式。
- 前端拆分模块时，共享 `provide/inject` key 必须保持运行时稳定；跨模块共享的 Symbol 使用 `Symbol.for(...)`，避免 Vite HMR、生成 JS 文件或重复模块实例导致注入失配白屏。
- 业务源码目录不要依赖生成的 `.js` 影子文件；若目录中已存在生成文件，导入解析和共享单例必须显式考虑 `.ts` / `.js` 并存风险。

## 自动化与发布要求

- 平台自动化优先稳定 DOM 选择器。
- 不使用固定屏幕坐标点击作为正式方案。
- 小红书发布要注意 Shadow DOM。
- 头条号自动发布要覆盖“投放广告赚收益”勾选逻辑。
- 发布失败必须保留日志、截图和错误原因。
- 不允许实现绕验证码、绕风控逻辑。

## 模型、Agent、Skills 要求

- 模型配置以页面显式配置为准，不依赖隐藏 `.env` 误导用户。
- Agent 只负责推理、总结、筛选、建议，不直接控制浏览器。
- Skills 是独立能力模块，只参与 prompt 约束，不直接操作账号登录态和发布任务。
- Agent 输出要尽量结构化，便于前后端稳定消费。

## 文档要求

- 修改功能时同步更新相关文档。
- 能力边界变化时同步更新 `docs/MODULE_BOUNDARY.md`。
- 阶段能力变化时同步更新 `docs/MVP_ROADMAP.md` 和 `docs/ai/` 下相关文档。
